# オンボーディング検証(成功基準2/3)

GOAL.md の成功基準のうち、コードでは自動判定できない2つを人手のユーザーテストで確かめる手順と記録。

- **成功基準2**: 形式手法未経験の開発者が、ドキュメントとサンプルだけで**1時間以内に最初の検査を回せる**
- **成功基準3**: 反例を見た開発者が、**助けなしにバグの筋書きを説明できる**

対象ドキュメントは [docs/tutorial.md](tutorial.md)、題材は [examples/tutorial-withdraw.ts](../examples/tutorial-withdraw.ts) になる。

## 被験者の条件

- Webサービス開発の実務経験があり、TypeScriptを読み書きできる
- 形式手法(TLA+ / Alloy / モデル検査)の使用経験がない
- このツールの開発に関わっていない(仕様・反例の内容を事前に知らない)

3名以上で実施する。1名だけでは、その人固有のつまずきと文書の欠陥を区別できない。

## 進め方

観察者は**質問に答えない**。詰まった箇所と経過時間を記録するだけにする。助けを出した時点で「助けなしに」の検証にならないため、完全に手が止まって5分経った場合のみ介入し、その回は「未達」として記録する。

被験者に渡すのはリポジトリのURLと次の一文だけにする。

> README を起点に、このツールで最初の検査を回してください。

### セッション1: 最初の検査を回すまで(成功基準2)

計測開始からの経過時間を、次の各時点で記録する。

| 通過点 | 記録 |
| --- | --- |
| T1 | SPAが起動してブラウザで開けた |
| T2 | ワンクリックデモの検査が完了し、反例が画面に出た |
| T3 | 仕様ファイルを自分で編集し、再検査して結果が変わるのを確認した |
| T4 | 自分で書いた(またはテンプレートから改変した)仕様の検査が通った |

**達成条件**: T4 が60分以内。

### セッション2: 反例の読み取り(成功基準3)

被験者にドキュメントを閉じてもらい、初見の反例を1つ見せる。`payment-retry`(リトライによる二重課金)を使う。tutorial-withdraw はセッション1で説明済みのため使わない。

反例を見せた上で、口頭で次を説明してもらう。

1. どの条件が破れたか
2. どういう順序で起きるとそうなるか(トレースの筋書きを自分の言葉で)
3. なぜそれが設計上まずいのか
4. どう直せば起きなくなるか

**達成条件**: 1〜3を助けなしに説明できる。4は設計判断が絡むため達成条件に含めず、参考として記録する。
