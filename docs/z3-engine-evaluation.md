# Z3(SMT)エンジン導入の評価と着手判断

データモデル・権限検証(フェーズ3)の検査エンジンは、`ModelEngine`インターフェースの背後で
差し替え可能に設計されている(`packages/spec/src/datamodel/engine.ts`、
[docs/datamodel-sketch.md](datamodel-sketch.md))。実装は全列挙エンジン(`enumerationEngine`)のみで、
Z3(WASM)ベースのエンジンは未実装。GOAL.mdの技術方針は「Z3ベースはまず小スコープ列挙で足りるか
検証してから着手する」という保留である。

本書は「小スコープの全列挙で足りるのか」「どの規模でZ3が必要になるのか」を実測を交えて評価し、
Z3実装への着手可否と着手条件(トリガー)を明示する意思決定ドキュメントである。

## 結論(先に)

- **現時点ではZ3を実装しない。** GOAL.mdの想定スコープ(`User: 2〜3`・`Doc: 1〜2`程度)は全列挙で
  余裕をもって完走でき、実行時間も1秒未満に収まる。小スコープ仮説(設計バグは小さいインスタンスで
  現れる)に立つ限り、現行エンジンで検出目的は達成できる。
- Z3は「検査空間が大きすぎて全列挙が打ち切られる」領域を扱うための投資であり、その領域に**実運用で
  常態的に踏み込むようになった時**に着手する。具体的なトリガーは[着手条件](#着手条件トリガー)に記す。
- 着手が必要になった場合の設計は[着手時の設計メモ](#着手時の設計メモ)に整理した。既存の`ModelEngine`
  インターフェースと`Formula`式木はZ3差し替えを見込んだ形になっており、モデル定義側の変更は不要。

## 全列挙エンジンのコスト構造

`enumerationEngine`は、各関係が取りうるタプル全体(引数ソートの直積)の**部分集合**を、関係ごとに
独立に選んでインスタンスを作る。したがって:

- 関係 `r` のタプル総数 `T(r)` = 引数ソートのスコープの積。
  例: 2項関係 `r(A, A)` で `A` のスコープが `n` なら `T(r) = n^2`。
- 関係 `r` の取りうる状態数 = そのタプル集合の部分集合数 = `2^T(r)`。
- モデル全体の総インスタンス数 = 全関係の総積 = `2^(ΣT(r))`。

つまり総インスタンス数は **スコープ `n` に対して二重指数的**(`2^(多項式(n))`)に増える。2項関係1本なら
`2^(n^2)`、2項関係2本なら`2^(2n^2)`。ソートを1つ増やす・アリティを1つ上げる・関係を1本足すたびに、
指数の肩(タプル総数)自体が跳ね上がる。

`maxInstances`(既定 `1_000_000`)を超えると列挙は打ち切られ、結果は `complete: false` になる。
打ち切られた場合、「その規模の反例が存在しないこと」は**保証されない**(未検査の空間に反例が
潜んでいるかもしれない)。

なお、この二重指数コストが顕在化するのは **反例が存在しない(性質が成り立つ)場合** である。反例が
あれば`findViolation`は最初に見つけたインスタンスで即座に返るため、実際の所要時間は総インスタンス数
より遥かに小さくなりうる。逆に「性質が正しいことを確認したい」ケースこそ全空間の走査が必要になり、
崖の影響を最も強く受ける。

## 実測データ

計測スクリプト: [packages/spec/bench/enumerationScaleBenchmark.ts](../packages/spec/bench/enumerationScaleBenchmark.ts)
(`npm run bench:enum -w @model-checking/spec`)。反例を出さない恒等式(番人述語)を`assertion`に置き、
`maxInstances`既定値(1,000,000)の下で全空間を走査させ、総インスタンス数(理論値・BigInt厳密)と
所要時間を測った。Node v24.12.0、開発機(WSL2上のLinux)での実測値。絶対値はマシン・負荷で変動する
ため、桁と「崖の位置」を読む。

### 単一の1項関係 r(A): 総数 = 2^n

| スコープ n | タプル総数 E | 総インスタンス数 2^E | 検査済み | complete | 経過時間 |
| --- | --- | --- | --- | --- | --- |
| 2 | 2 | 4 | 4 | true | 0.4 ms |
| 4 | 4 | 16 | 16 | true | 0.1 ms |
| 8 | 8 | 256 | 256 | true | 2.4 ms |
| 16 | 16 | 65,536 | 65,536 | true | 231 ms |
| 20 | 20 | 1,048,576 | 1,000,000 | **false** | 3,798 ms |
| 24 | 24 | 16,777,216 | 1,000,000 | **false** | 4,436 ms |

1項関係は緩やか(`2^n`)。崖は `n = 20`(`2^20 ≈ 1.05M`)で初めて現れる。

### 単一の2項関係 r(A,A): 総数 = 2^(n^2)

| スコープ n | タプル総数 E | 総インスタンス数 2^E | 検査済み | complete | 経過時間 |
| --- | --- | --- | --- | --- | --- |
| 2 | 4 | 16 | 16 | true | 0.1 ms |
| 3 | 9 | 512 | 512 | true | 0.9 ms |
| 4 | 16 | 65,536 | 65,536 | true | 204 ms |
| 5 | 25 | 33,554,432 | 1,000,000 | **false** | 4,840 ms |
| 6 | 36 | 68,719,476,736 | 1,000,000 | **false** | 6,633 ms |

2項関係1本は `2^(n^2)`。`n = 4` までは完走(65,536)、**`n = 5` で崖**(`2^25 ≈ 33.5M`、1Mの33倍)。
`n = 3 → 4 → 5` で総数は `512 → 65,536 → 3,355万` と桁が跳ね上がる。

### 2本の2項関係 r(A,A), s(A,A): 総数 = 2^(2n^2)

| スコープ n | タプル総数 E | 総インスタンス数 2^E | 検査済み | complete | 経過時間 |
| --- | --- | --- | --- | --- | --- |
| 2 | 8 | 256 | 256 | true | 1.0 ms |
| 3 | 18 | 262,144 | 262,144 | true | 512 ms |
| 4 | 32 | 4,294,967,296 | 1,000,000 | **false** | 3,754 ms |

関係が2本になると指数の肩が倍。`n = 3` までは完走(262,144)、**`n = 4` で崖**(`2^32 ≈ 43億`)。

### ドキュメント権限モデル(4関係): 総数 = 2^(3·U·D + U)

題材モデル([examples/doc-permission.ts](../examples/doc-permission.ts))と同じ関係構成
(`owner[U,D]`, `sharedWith[U,D]`, `admin[U]`, `canEdit[U,D]`)。タプル総数 = `3·U·D + U`。

| User U | Doc D | タプル総数 E | 総インスタンス数 2^E | 検査済み | complete | 経過時間 |
| --- | --- | --- | --- | --- | --- | --- |
| 2 | 1 | 8 | 256 | 256 | true | 0.6 ms |
| 3 | 1 | 12 | 4,096 | 4,096 | true | 8.8 ms |
| 2 | 2 | 14 | 16,384 | 16,384 | true | 21.5 ms |
| 4 | 1 | 16 | 65,536 | 65,536 | true | 102 ms |
| 3 | 2 | 21 | 2,097,152 | 1,000,000 | **false** | 1,643 ms |
| 2 | 3 | 20 | 1,048,576 | 1,000,000 | **false** | 1,628 ms |
| 5 | 1 | 20 | 1,048,576 | 1,000,000 | **false** | 1,885 ms |

`examples/doc-permission.ts`の既定 `User: 2, Doc: 1`(256インスタンス、0.6ms)は崖から十分に遠い。
題材の反例は`User: 1, Doc: 1`でも出るため、実運用上まったく問題ない。一方、**`User: 3, Doc: 2` や
`Doc: 3`、`User: 5` に上げると崖**(1Mを超え打ち切り)に達する。

## 小スコープ全列挙で足りるか

**足りる範囲(現行エンジンで完全検証できる)**:

- 2項関係が主体のモデルで、各ソートのスコープが概ね `n ≤ 4`、関係が1〜2本程度。
- ドキュメント権限モデル型(4関係、うち2項3本+1項1本)で、`3·U·D + U ≤ 20` を目安に
  `User ≤ 4, Doc = 1` あるいは `User ≤ 2, Doc ≤ 2`。
- 1項関係主体(権限フラグの列挙など)なら `n ≤ 19` まで完走。

この範囲は、GOAL.mdが想定する小スコープ(`User: 2〜3`・`Doc: 1〜2`)と、小スコープ仮説
(設計バグは小さいインスタンスで現れる。TLA+/Alloy実務の知見)を十分にカバーする。**権限モデルの
抜け漏れ・スキーマ制約の破れといった設計バグの検出という目的に対して、現行の全列挙で足りる。**

**足りなくなる範囲(崖に当たる)**:

- 2項関係1本で `n ≥ 5`、2本で `n ≥ 4`。
- 3項以上の関係(`T(r) = n^3`。`n = 3` で既に `2^27`)。現状の式木の語彙でも3項関係は表現可能で、
  スコープ3程度で即座に崖に達する。
- 上記が組み合わさり、必要スコープで `ΣT(r) > 20` になるモデル全般。

この範囲では `complete: false` となり、「反例が無い」ことを主張できない(部分検査に留まる)。ただし
小スコープ仮説に立つ限り、そもそもこの範囲まで上げる必要が生じるのは例外的である。

## Z3導入で解決すること / しないこと

**解決すること**:

- **崖の解消**。SMTソルバは部分集合を1つずつ列挙せず、制約充足問題として反例の有無を判定する。
  検査コストは総インスタンス数(`2^(ΣT(r))`)に線形には依存せず、スコープを上げても全列挙のように
  二重指数で即死しない(理論上はNP困難だが、実用上は現実的なスコープで解けることが多い)。
- **より大きいスコープでの完全性**。`complete: false` の打ち切りに頼らず「このスコープに反例は無い」を
  ソルバの`unsat`として結論できる範囲が広がる。
- **算術・集合濃度など語彙拡張との相性**(#16で式木に算術・濃度を追加予定)。全列挙は数値ソートの
  スコープを直積で持つため数値レンジに弱いが、Z3は整数・線形算術を得意とする。

**解決しないこと**:

- **小スコープ仮説そのものは変わらない**。Z3を入れても「検証すべき性質は小さい反例を持つ」という前提の
  上に立つ設計は同じ。Z3は「小スコープでは足りず中〜大スコープの完全性が要る」場面のための道具であって、
  検出できるバグの種類を根本的に変えるものではない。GOAL.mdの非ゴール(大規模状態空間の網羅検査、
  定理証明)は依然として非ゴール。
- **無限領域・完全な汎用証明**。量化子を含む一階論理はZ3でも一般には決定不能で、量化子の扱いによっては
  `unknown` を返しうる。有限スコープの範囲を出る保証にはならない。
- **導入コストの増加**。WASMバイナリ(z3-solver等)のバンドルサイズ、Worker統合、式木→SMT翻訳と
  反例の逆変換という新たな実装・保守対象が増える。GOAL.mdの「WASM不使用(純JS/TS)」方針の唯一の
  例外として、その複雑さに見合う必要が生じてから払うべきコストである。

## 着手条件(トリガー)

以下のいずれかが**実運用で常態化した**ら、Z3エンジン実装に着手する。単発・実験的に崖へ触れただけでは
着手しない(小スコープ仮説に立てば通常は崖の手前で足りるため)。

1. **打ち切りの常態化**: 実チームの実モデルを検証する際、検出に必要なスコープで総インスタンス数が
   `10^6` を超え、`complete: false`(打ち切り)が繰り返し発生する。すなわち「反例が無いことを確認
   したいのに、そのスコープを全列挙し切れない」状況が定常的に起きる。
2. **特定性質が現行で検証不能**: 3項以上の関係や、複数の2項関係の組み合わせが本質的に必要な性質が
   現れ、必要スコープ(`ΣT(r) > 20` 目安)を全列挙では到底さばけない。
3. **語彙拡張(#16)による要求**: 算術・集合濃度を式木に加えた結果、数値ソートのレンジを直積で持つ
   全列挙が非現実的になり(数値レンジで即座に崖)、SMTの線形算術が実質的に必須になる。
4. **完全性への要件**: レビュー用途で「このスコープに反例は無い」を根拠づけたいのに、対象モデルの
   必要スコープが全列挙の完走域を超えており、部分検査(`complete: false`)では意思決定に足りない、
   という要求が繰り返し挙がる。

補助指標として、`bench:enum` の数値表で対象モデルの構成(関係のアリティ・本数)における崖スコープを
見積もり、想定運用スコープがそれを超えるなら着手検討の材料とする。

## 着手時の設計メモ

Z3エンジンは、既存の設計にそのまま差し込める形を想定している。

### エンジンの差し込み

- `ModelEngine`(`findViolation(model, options): ModelCheckResult`)を実装する `z3Engine` を新設する。
  `enumerationEngine` と同じインターフェース・同じ `ModelCheckResult` を返すため、`checkModel` が使う
  エンジンを切り替えるだけでよい(モデル定義 `ModelDef`・式木 `Formula` は無変更)。
- `ModelCheckResult` の `ok: false` 側は `{ assertion, instance, instancesChecked }`。Z3は反例を
  `instance`(`Instance` 型: ソート→原子名、関係→タプル集合)へ逆変換して詰める。`ok: true` 側の
  `instancesChecked` / `satisfiedInstances` は全列挙前提の指標なので、Z3では意味を再定義するか
  (例: `sat/unsat` の別フィールド)、`ModelCheckResult` を判別可能に拡張する。

### 式木 → SMT-LIB / Z3 API 翻訳

各ソートを有限の未解釈ソート(または列挙型/有限domainの整数)として宣言し、関係を述語関数
(`(Sort... ) -> Bool`)として宣言する。`Formula` の各ケースは以下へ対応する:

| Formula ケース | Z3構成子 |
| --- | --- |
| `forall(sort, x => body)` | `ForAll([x], body)`(`x` はそのソートの束縛変数) |
| `exists(sort, x => body)` | `Exists([x], body)` |
| `rel(name, ...args)` | 述語関数適用 `name(args...)` |
| `eq(a, b)` / `neq(a, b)` | `a == b` / `Not(a == b)` |
| `and` / `or` / `not` | `And(...)` / `Or(...)` / `Not(...)` |
| `implies` / `iff` | `Implies(l, r)` / `l == r`(Bool同値) |

検査は「`constraints` を全て `assert` し、`assertion` の**否定**を `assert` して `check-sat`」。
`sat` なら反例(モデル)が存在=性質違反、`unsat` なら反例なし=性質成立。複数 `assertion` は
1本ずつ検査する(全列挙が最初の違反で返るのと対応)。スコープは有限化のため、各ソートに
`|sort| = scope` の濃度制約(相異なる原子の存在+全要素がそのいずれか)を課すか、有限列挙ソートを使う。

`de Bruijn` ではなく一意 `varId` を振る現在の式木表現(`formula.ts`)は、翻訳時に `varId → Z3の束縛変数`
のマップを持てばそのまま扱える。

### 反例(model)→ Instance への逆変換

`sat` のとき、Z3のモデルから各ソートの原子集合と各関係の真になるタプルを読み出し、`Instance`
(`atoms` / `relations`)へ復元する。可視化(apps/web)は `Instance` を「ソートごとの原子一覧」
「関係ごとのタプル表」としてそのまま描画するため、逆変換さえ合わせれば表示側の変更は不要。

### WASM / ブラウザ・Worker 統合

- `z3-solver`(Z3公式のWASMビルド、npm配布)を候補とする。GOAL.mdの「検査はWeb Worker上で実行」
  「WASM不使用の例外はZ3(ソルバ内で計算が完結するためWASMが効く)」の方針に沿う。
- バンドルサイズ(数MB規模のwasm)とロード時間を考慮し、Z3エンジンは遅延ロード(必要になった時だけ
  Workerで初期化)する。SPA(静的ホスティング・データを外部送信しない)という配布制約は、wasmを
  同梱すれば維持できる。
- ソルバ呼び出しは非同期(wasm初期化・`check-sat`)になるため、`ModelEngine.findViolation` の同期
  シグネチャと整合させる必要がある。Worker境界は元々非同期なので、Worker内で完結させるか、
  エンジンインターフェースに非同期版を追加するかを実装時に決める。
