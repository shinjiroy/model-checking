# ウォッチモードの実ファイルシステム動作検証

ウォッチモード(フォルダ選択→保存→自動再検査ループ)は、フォルダ選択に File System Access API の
`showDirectoryPicker` を使う。このピッカーはブラウザのセキュリティ上、ヘッドレス自動操作(Playwright 等)から
起動できないため、一連の実フォルダ操作は自動 E2E テストの対象外となる。純ロジック(スナップショット比較・
差分判定・セッションガード・ポーリング制御)は以下の自動テストで担保している。

| 層 | テスト |
| --- | --- |
| スナップショット比較・差分・全件再読み込み([core/watch.ts](../apps/web/src/core/watch.ts)) | [tests/watch.test.ts](../apps/web/tests/watch.test.ts) |
| ポーリング・セッションガード([ui/useDirectoryWatch.ts](../apps/web/src/ui/useDirectoryWatch.ts)) | [tests/useDirectoryWatch.test.tsx](../apps/web/tests/useDirectoryWatch.test.tsx) |
| App への配線(保存検知→自動再検査)([App.tsx](../apps/web/src/App.tsx)) | [tests/App.watch.test.tsx](../apps/web/tests/App.watch.test.tsx) |

自動テストでカバーできない「実ブラウザで実フォルダを選び、実際のファイル保存を検知し、再検査が走り、停止できる」
一連の経路は、本書の手順で手動確認する。

## 前提

- **ブラウザ**: File System Access API 対応の Chrome または Edge(Chromium 系)。Firefox / Safari では
  「フォルダを開いて監視」ボタン自体が表示されず(ドラッグ&ドロップにフォールバック)、本検証の対象外。
- **サーバー**: 開発サーバーを起動しておく。

  ```bash
  docker compose up -d web    # http://localhost:5173
  # または
  npm install && npm run -w @model-checking/web dev
  ```

- **検証用フォルダ**: [examples/](../examples/) を検証対象フォルダとして使う。中に `.ts` の仕様ファイルが
  複数あり、`node_modules` や `.` 始まりのディレクトリは監視対象から除外される。

## 監視対象の仕様(挙動の前提)

- 監視対象は `.ts` / `.tsx` ファイルのみ。`node_modules` と `.` 始まりディレクトリは再帰的に除外する。
- ポーリング間隔は **1 秒**。各ファイルの `lastModified` の変化だけで検知し、変化があれば対象フォルダの
  ファイル一式を読み直す。
- フォルダを開いた直後は初期スナップショットを取るだけで再検査は走らない(開いた時点の内容はすでに解析経路に
  乗っているため)。以降の**保存(変更・追加・削除)**で再検査が走る。
- フォルダを開く直前に spec を選んでいれば、保存検知後の再解析に続けて**同じ spec の検査まで自動実行**される
  (`autoCheck`)。spec 未選択なら解析までで止まる。

## 検証手順

各項目の「期待」をすべて満たせば合格。

### 1. フォルダ選択

1. アプリを開く(`http://localhost:5173`)。
2. 「ウォッチモード」セクションの **「フォルダを開いて監視」** をクリックする。
3. OS のフォルダ選択ダイアログでリポジトリの `examples/` を選び、閲覧を許可する。

**期待**:
- 「監視中: **examples**(保存すると自動で再検査します)」と表示され、「監視を停止」ボタンが出る。
- `examples/` のファイルが読み込まれ、spec の選択肢(`order-payment` / `payment-retry` / `doc-permission`)が
  出る。
- ダイアログでキャンセルした場合は何も変化せず、エラーも出ない。

### 2. 保存検知 → 自動再検査(spec 選択あり)

1. 手順 1 に続けて、spec を 1 つ選び **「検査する」** を実行し、結果(反例トレース等)が出ることを確認する。
2. エディタで [examples/payment-retry.ts](../examples/payment-retry.ts) を開き、コメント追記など無害な変更を
   加えて**保存**する。

**期待**:
- 保存から約 1 秒以内に自動で再解析→**再検査**まで走る(検査中は探索済み状態数が更新される)。
- 選んでいた spec の検査結果が改めて表示される。
- 手動で「解析する」「検査する」を押す必要がない。

### 3. 追加・削除の検知

1. `examples/` に新しい `.ts` ファイル(例: `tmp-check.ts`、中身は既存 spec のコピーで可)を**追加保存**する。
2. 期待を確認したら、その `tmp-check.ts` を**削除**する。

**期待**:
- 追加時: 約 1 秒以内に再読み込みが走り、spec 選択肢に追加分が反映される。
- 削除時: 約 1 秒以内に再読み込みが走り、選択肢から消える。
- `node_modules` 配下や `.` 始まりのファイルを触っても再検査は走らない(監視対象外)。

### 4. 監視停止

1. **「監視を停止」** をクリックする。
2. 停止後、監視していたファイルをもう一度**保存**する。

**期待**:
- ボタンが「フォルダを開いて監視」に戻り、「監視中」表示が消える。
- 停止後の保存では**再検査が走らない**(ポーリングが止まり、停止直前に進行中だったポーリングの結果も
  セッションガードで破棄される)。

### 5. 別ソース読み込みによる暗黙停止

1. 再度フォルダを開いて監視中にする。
2. 監視を明示的に止めず、デモ仕様の読み込みボタンやドラッグ&ドロップで**別のソースを読み込む**。

**期待**:
- 「監視中」表示が消える(別ソース読み込み時に監視は自動停止する)。
- 元フォルダを保存しても再検査は走らない。

### 6. アクセス拒否・エラー表示

1. フォルダ選択ダイアログで閲覧許可を拒否する、またはアクセス権のないフォルダを選ぶ。

**期待**:
- 「フォルダへのアクセスが拒否されました」等のエラーメッセージが表示され、アプリはクラッシュしない。
- 監視状態には入らない。

## 記録

検証時は以下を控える(不具合報告時の再現条件になる)。

- ブラウザ名とバージョン(例: Chrome 126 / Edge 126)
- OS
- 各手順の合否
- 不合格があれば、操作と実際の挙動・スクリーンショット
