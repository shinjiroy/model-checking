# 設計方針

このツールが何を目指し、何を目指さないか。個々の実装判断の前提になる。

## ゴール

**Webサービス開発チームの一般的な開発者が、形式手法の事前知識なしに、設計上の欠陥を実装前に発見できるブラウザ完結型の検証ツール。**

- 仕様は **TypeScript** で書く。新言語は作らず、状態・アクション・不変条件をTSのオブジェクト/関数で記述する
- 検査(状態空間探索)は **ブラウザ内で完結** する(SPA、サーバーなし、Web Worker上で実行)
- 反例は **人間が読める形**(タイムライン・状態遷移の再生・シーケンス図)で可視化する
- 仕様と結果は **URLで共有** でき、設計レビューでそのまま使える

## 想定利用者

チームの一般的な開発者(形式手法の経験なし)。学習コスト最小化が最優先で、仕様がTypeScriptなので既存のエディタ・型チェック・レビューフローがそのまま使える。エラーメッセージと反例の可視化が生命線になる。fast-checkやXStateの経験から地続きで理解できる設計にする。

## 検証対象

| 対象 | 検出したい欠陥の例 | 検査方式 |
| --- | --- | --- |
| 状態遷移・業務フロー | フローのデッドロック、到達不能状態、不変条件違反 | 明示的状態探索(BFS) |
| 並行・分散の整合性 | 非同期API・リトライ・排他制御のタイミング依存バグ | 複数プロセスのインターリーブ探索 |
| データモデル・権限 | 権限モデルの矛盾・抜け漏れ、スキーマ制約の破れ | 小スコープの全インスタンス列挙+制約チェック |

データモデル側が小スコープの列挙で足りるのは**小スコープ仮説**による。設計上のバグは小さいインスタンスで現れる、という経験則で、Alloyが依拠しているのと同じ前提にあたる。

## 非ゴール

- **大規模状態空間(数千万状態〜)の検査** — 小さいモデルで設計バグの検出には十分、というのがTLA+実務の知見
- **実装コードとの適合性検証(refinement)**
- **定理証明・時相論理の完全サポート** — 安全性(不変条件・デッドロック)を主体にする。活性(liveness)は必要になってから判断する
- **TLA+ / Alloy との互換性**

## 技術方針

### 仕様記述

TypeScript埋め込みDSL。`{ init, actions, invariants }` で状態機械を定義する。

- 状態遷移側は素のTS関数で書く(実行して評価するだけ)
- データモデル側の述語はコンビネータ(`forall` / `exists` / `eq` 等)の式木で書く。boolean関数はSMT式に翻訳できないため、全列挙とZ3の両方で解釈できるASTを挟み、仕様を書き換えずにエンジンを差し替えられるようにする。述語の語彙は式木の範囲(量化・等値・関係演算・算術程度)に制限されるが、これはZ3の表現範囲とほぼ一致する

### 検査器

TS自作の明示的状態探索器。Web Worker上で実行する。重複排除により数十万〜数百万状態を目標にする。

### WASM不使用(純JS/TS)

探索のホットループはユーザーのTS関数呼び出しであり、検査器をWASM化しても1状態ごとの境界往復で利得が消える。性能の勘所はデータ構造の側にある。

- 訪問済み状態は64bitフィンガープリントでTypedArrayに保持する(100万状態で約8MB。WASM線形メモリと同等のメモリモデル)
- 状態はGC圧を避けるコンパクトな表現にする
- マルチコアはWorkerのハッシュ分割で使う

例外はZ3で、ソルバ内で計算が完結するためWASMが効く。

### 仕様の入力

TSファイル群(複数ファイル・フォルダ)をドラッグ&ドロップまたはFile System Access APIで受け取り、esbuild-wasmでブラウザ内バンドルしてWorkerで実行する。

- importはDSLパッケージ(エイリアス注入)と相対パスのみ。ブラウザで解決できないうえ、外部ライブラリを排除することが状態のシリアライズ可能性の担保にもなる
- ブラウザは型チェックせずトランスパイルのみを行う。型チェックはDSLをnpm配布して手元のエディタ・CIに寄せる([spec-package.md](spec-package.md))
- Chrome/Edge系は保存→自動再検査のウォッチモード、他のブラウザはドラッグ&ドロップにフォールバックする

### 配布

SPA(静的ホスティングのみ)。仕様データを外部に送らない。URL共有はバンドル後ではなくソース全ファイルを圧縮してフラグメントに載せる(開いた側で編集を続けられるため)。詳細は [deployment.md](deployment.md)。

## 設計上の制約

実装を変えるときも守るもの。

- サーバーを立てない
- 検査器コアはWASM化しない
- トレース形式(`CheckResult` / `TraceStep`)は可視化との契約であり、変更時は [dsl-sketch.md](dsl-sketch.md) と [trace-visualization.md](trace-visualization.md) を同時に更新する

## 検討して採らなかった選択肢

| 案 | 採らなかった理由 |
| --- | --- |
| TLC / Alloy のブラウザ移植 | Java製で移植が重く、言語の学習コストも残る |
| Quint | TS製ツール群だが独自構文の学習が必要 |
| 新言語のフル設計 | 失敗リスクと工数が最大 |
| Z3ベースの検査 | まず小スコープ列挙で足りるかを検証してから判断する。現時点の結論は [z3-engine-evaluation.md](z3-engine-evaluation.md) |
