# チュートリアル: 最初の1仕様を書いて検査する

形式手法の事前知識は要らない。TypeScriptが書ければ読める。所要時間はおよそ40分。

このチュートリアルを終えると、次ができるようになる。

- ワンクリックデモで検査を回し、反例トレースを読む
- 自分の設計を「状態・アクション・不変条件」の3つに分けて書く
- 見つかった反例からバグの筋書きを説明し、仕様を直して反例が消えることを確かめる

## このツールがしていること

設計を状態機械として書くと、検査器が**初期状態から到達できる状態を全部**たどる。各状態で発火できるアクションを1つ残らず試すので、「AとBが同時に動いたときのすべての順序」が自動的に網羅される。

その途中で「成り立っていてほしい条件(不変条件)」が破れる状態に着き当たると、そこまでの手順を**反例トレース**として返す。探索は幅優先なので、返るトレースは常に**最短**の手順になる。

人間のレビューが苦手なのは、この「ありうる順序の総当たり」にあたる。テストは書いた順序しか試さないが、検査器は書かなかった順序を試す。

## 0. 起動する(5分)

```bash
docker compose up -d web    # http://localhost:5173
```

Dockerを使わない場合:

```bash
npm install && npm run -w @model-checking/web dev
```

ブラウザで http://localhost:5173 を開く。仕様はブラウザ内でバンドル・検査され、サーバーへは送信されない。

## 1. 反例を見る(10分)

「tutorial-withdraw(入門: 残高がマイナスになる)」のワンクリックデモを読み込み、「解析する」→「検査する」を押す。

反例が出る。これは次の設計を検査したものになっている。

> 残高100円の口座に、60円を引き出す処理が2つ同時に走る。それぞれ「残高が足りるか確認する」→「引き落とす」の順に動く。

タイムラインを左から読むと、`処理A` と `処理B` のレーンに4ステップが並んでいる。

| # | アクション | 起きること |
| --- | --- | --- |
| 1 | `checkA` | 処理Aが残高を確認する。100 ≥ 60 なので引き落として良いと判断する |
| 2 | `checkB` | 処理Bも残高を確認する。まだ誰も引き落としていないので 100 ≥ 60、こちらも引き落として良いと判断する |
| 3 | `withdrawA` | 処理Aが引き落とす。残高は 40 になる |
| 4 | `withdrawB` | 処理Bが引き落とす。**残高は -20 になる** |

不変条件 `balanceNeverNegative`(残高はマイナスにならない)が破れた。ステップを選ぶと直前の状態との差分が出るので、`balance` がどこで負に落ちたかを目で追える。

バグの筋書きは「**確認した時点の残高が、引き落とす時点でも同じだと思い込んでいる**」。確認と引き落としの間に他人の引き落としが挟まりうる。よくある time-of-check to time-of-use の取り違えにあたる。

このトレースを見ずに、コードレビューだけでこの順序を思いつけるかを考えてみてほしい。それがこのツールの効き目になる。

## 2. 仕様を読む(10分)

[examples/tutorial-withdraw.ts](../examples/tutorial-withdraw.ts) が今検査した仕様そのものになっている。仕様は3つの部品でできている。

### 状態: 何が変わりうるか

設計の中で時間とともに変わるものを、JSONにできるプレーンオブジェクトとして並べる。

```typescript
type State = {
  balance: number;    // 口座残高
  checkedA: boolean;  // 処理Aが残高確認を済ませたか
  checkedB: boolean;
  doneA: boolean;     // 処理Aが終了したか
  doneB: boolean;
};
```

「確認したか」を状態に持たせているのが要点になる。確認と引き落としを別々のアクションに分けるからこそ、検査器が**その間に割り込む**順序を試せる。1つのアクションにまとめて書くと、それは「確認と引き落としが原子的に起きる」という別の設計を書いたことになり、このバグは出てこない。

**どこまで細かく分けるか**が、モデル化のほぼ唯一の判断になる。目安は「間に他の処理が割り込みうるか」。割り込みうるなら分ける。

### アクション: どう変わるか

`when`(発火できる条件)と `then`(発火したときの新しい状態)の組で書く。`then` は元の状態を書き換えず、新しいオブジェクトを返す純粋関数にする。

```typescript
withdrawA: {
  actor: "処理A",                          // 可視化のレーン分けに使う。検査結果には影響しない
  when: s => s.checkedA && !s.doneA,       // 確認済みでまだ引き落としていないなら発火できる
  then: s => ({ ...s, balance: s.balance - AMOUNT, doneA: true }),
},
```

検査器は各状態で `when` が真のアクションを**全部**試す。「どちらが先か」を書く必要はない。書かないことが「どの順序でも起きうる」を意味する。

### 不変条件: 何が成り立っていてほしいか

```typescript
invariants: {
  balanceNeverNegative: s => s.balance >= 0,
},
```

「バグとはどういう状態か」ではなく「**正しいとはどういう状態か**」を書く。ここが設計意図の置き場になる。

もう1つ `done` がある。

```typescript
done: s => s.doneA && s.doneB,
```

検査器は「発火できるアクションが1つもない状態」をデッドロックとして報告する。`done` が真の状態は「正常に終わった」とみなして報告から除く。

## 3. 直して、反例が消えるのを確かめる(5分)

引き落とす直前にもう一度残高を確認するように `withdrawA` / `withdrawB` を直す。

```typescript
withdrawA: {
  actor: "処理A",
  when: s => s.checkedA && !s.doneA,
  then: s =>
    s.balance >= AMOUNT
      ? { ...s, balance: s.balance - AMOUNT, doneA: true }
      : { ...s, doneA: true },   // 足りなければ引き落とさずに終わる
},
```

エディタで [examples/tutorial-withdraw.ts](../examples/tutorial-withdraw.ts) をこう書き換えて保存し、SPAに読み込み直して検査すると、反例が消えて「検査に成功」になる。探索した状態数も表示される。

Chrome/Edge系なら「フォルダを開いて監視」でウォッチモードにしておくと、保存するたびに自動で再検査される。

「反例が出る」と「直すと消える」の両方を回帰テストに固定してあるのが [packages/spec/tests/tutorial-withdraw.test.ts](../packages/spec/tests/tutorial-withdraw.test.ts) になる。`npm test` で走る。

## 4. 自分の設計を書く(10分)

新しい `.ts` ファイルを作り、`defineSpec` に3つの部品を埋める。

```typescript
import { defineSpec } from "@model-checking/spec";

type State = {
  // 時間とともに変わるものを並べる
};

export const mySpec = defineSpec<State>({
  init: {
    // 最初の状態
  },
  actions: {
    myAction: {
      actor: "誰が",
      when: s => true,
      then: s => ({ ...s }),
    },
  },
  invariants: {
    myInvariant: s => true,
  },
  done: s => false,
});
```

`export` した `defineSpec` の戻り値をSPAが見つけて検査する(1ファイルに複数あると選択できる)。

### 題材の選び方

このツールが最も効くのは、**複数のものが同時に動く**設計になる。

- 非同期API・Webhook・メッセージキュー(応答の到着順が決まらない)
- リトライ・タイムアウト(同じ要求が二度届きうる)
- 排他制御・在庫や残高の更新(read-modify-write が競合する)
- 複数ユーザーの同時操作

逆に、単独プロセスの逐次処理には競合の反例がないので、効きは薄い。

### 最初に踏みやすい点

- **状態空間を有界にする**: 配列が無限に伸びる仕様(リトライ回数に上限がない等)は上限で打ち切られ、「打ち切り」表示になる。回数の上限や重複防止フラグで有界にする
- **`then` で元の状態を書き換えない**: 検査器は状態を凍結するので、破壊的変更は例外として即座に出る。必ず `{ ...s, ... }` で新しいオブジェクトを返す
- **状態はJSONにできるものだけ**: `Map`・`Set`・クラスインスタンス・関数は使えない。配列とプレーンオブジェクトで表す
- **小さく始める**: ユーザー2人・リトライ2回で十分に反例は出る。大きくするのは反例が出なくなってからで良い
